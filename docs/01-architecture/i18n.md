# i18n

## Enfoque
El frontend usa `next-intl` en App Router sin `i18n routing`.

Eso significa:
- no existen rutas `/es/...` o `/en/...`
- las rutas siguen siendo `/login`, `/signup`, `/app/*`
- el idioma activo se decide por la cookie `bs_locale`

## Fuente de verdad
La cookie de locale es:
- `bs_locale`

Reglas:
- default: `es`
- permitidos: `es`, `en`
- si la cookie falta o es inválida, el request cae a `es`

La configuración vive en:
- `frontend/src/i18n/config.ts`
- `frontend/src/i18n/request.ts`

## Messages
Los catálogos viven en:
- `frontend/src/i18n/messages/es.json`
- `frontend/src/i18n/messages/en.json`

Convención:
- keys estables por dominio
- namespaces cortos: `common`, `nav`, `auth`, `ui`, `dashboard`, `services`, `clients`, `placeholders`

Para agregar nuevas traducciones:
1. agrega la key en `es.json`
2. agrega la misma key en `en.json`
3. usa `getTranslations()` en Server Components o `useTranslations()` en Client Components

## Provider global
El layout raíz envuelve la app con `NextIntlClientProvider` en:
- `frontend/src/app/layout.tsx`

Así quedan disponibles traducciones tanto en:
- Server Components
- Client Components

## Cambio de idioma
El cambio de idioma se hace sin navegar a otra ruta.

Flujo:
1. `LanguageToggle` hace `POST /api/i18n/locale`
2. el Route Handler valida `es | en`
3. setea `bs_locale`
4. el cliente ejecuta `router.refresh()`
5. los Server Components se re-renderizan con el nuevo locale

Archivos:
- `frontend/src/components/language-toggle.tsx`
- `frontend/src/app/api/i18n/locale/route.ts`

## Dónde se usa hoy
La infraestructura ya cubre el copy visible más importante:
- app shell y navegación
- login y signup
- dashboard
- services
- clients
- placeholders del shell

## Nota futura
Si más adelante hace falta locale en URL, `next-intl` soporta routing con middleware/proxy.

Hoy no lo hacemos para:
- no romper rutas existentes
- no romper BFF, CSRF ni E2E
- mantener URLs limpias para demo
